Práctica 2: Cliente-Servidor mediante pipes

Situación inicial

En esta práctica se intenta realizar un sencillo sistema cliente-servidor utilizando pipes (tuberías) de Unix. El grupo de prácticas deberá entregar un código fuente, con su documentación asociada, que cumpla los requisitos que se proponen en el enunciado.

Enunciado

El grupo se comportará como un equipo de desarrollo, entregando al profesor, que actuará como cliente, el código fuente y los manuales de instalación y de usuario de dicho código. El grupo podrá elegir uno de los dos posibles aplicaciones que se muestran a continuación. Sea cual sea la aplicación elegida, deberá poderse ejecutar mediante el comando make. Los procesos cliente y servidor de la aplicación deberán contener en sus nombres las cadenas cli2 y serv2 respectivamente. Dichos nombres pueden comprobarse mediante el comando ps de Unix. En el caso de que la aplicación se realice mediante un único ejecutable, éste se llamará cliserv2 pero deberá generar un mínimo de dos procesos cuyos nombre deberían ser cli2 y serv2.

Aplicación 1: cliente-servidor de ficheros

Se trata de realizar una sencillo caso de aplicación cliente-servidor de ficheros. El cliente se conectará al servidor mediante una tubería (pipe) y le enviará el path completo de un fichero. El servidor responderá con los contenidos de dicho fichero, o un mensaje de error si no pudiera proporcionárselos. El cliente mostrará la respuesta recibida por el servidor en su salida standard.

Aplicación 2: cliente-servidor de fecha y hora

De nuevo un caso sencillo de aplicación cliente-servidor. El cliente se conectará al servidor mediante una tubería (pipe), momento en el cual el servidor le contestará con una cadena de caracteres que represente la fecha y la hora (con precisión mínima de 1 segundo) en el formato standard internacional asociado a la localización geográfica (país y zona horaria) del sistema en el que se ejecuta el servidor. El cliente mostrará la respuesta recibida por el servidor en su salida standard.

Requisitos obligatorios

* En las dos aplicaciones ambos procesos terminarán una vez realizada una interacción básica; no es necesario que se ejecuten en bucle. * Tampoco es necesario que sean capaces de atender a más de un cliente simultáneamente.

Requisitos opcionales

* Permitir la conexión de más de un cliente al servidor simultáneamente (implica que el servidor no debe terminar tras atender a un cliente). * Que el servidor permita la conexión de clientes de otros grupos de prácticas. * Que el cliente sea capaz de conectarse a servidores de otros grupos de prácticas. * Funcionar en otro sistema operativo que disponga de pipes. * Que tanto el cliente como el servidor sean configurables, en particular en lo tocante al nombrado necesario para las primitivas de comunicación entre procesos. * Documentación opcional:
  • Documento de arquitectura
  • Documento de diseño
  • Manual de administración
* Realización de las dos aplicaciones.
